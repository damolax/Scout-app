SCOUT v10.35.1 SCALE GUARD — R3 ORPHAN-SENDER COMPATIBILITY RELEASE

WHY R3 EXISTS
The live sent_messages table contains historical rows whose gmail_account_id points
to a Gmail account that was later disconnected or deleted. Older Scout releases did
not enforce a foreign key on sent_messages.gmail_account_id, so those historical rows
are valid history but cannot be inserted into a new active-account summary table with
a foreign key.

R3 does not delete or rewrite those historical sent_messages rows. It skips only the
orphaned account references while building scout_sender_lifetime_stats. It also guards
the future lifetime-stat trigger so a stale account reference cannot break a send-row
insert.

IMPORTANT
- Do not run the R2 SQL again.
- Use RUN_THIS_SQL_FIRST_V10_35_1_SCALE_GUARD_R3.sql.
- The R3 SQL is additive and idempotent. It is safe to run after either failed R2
  attempt, because PostgreSQL DDL and CREATE/ALTER statements are repeated safely.
- The application has not been pushed by the earlier failed deployment script.

STEP 1 — BACK UP SUPABASE
Create a database backup or export.

STEP 2 — RUN THE COMPLETE R3 SQL
Run:
RUN_THIS_SQL_FIRST_V10_35_1_SCALE_GUARD_R3.sql

A NOTICE may report that historical sent-message rows were skipped because their
Gmail account no longer exists. That is expected. The sent-message history remains.
The query must finish with Success and no ERROR.

STEP 3 — KEEP/CONFIRM VERCEL VARIABLES
SCHEDULE_WORKER_SECRET=CREATE_ONE_LONG_RANDOM_SECRET
SCOUT_MAX_ACTIVE_CAMPAIGNS=12
SCOUT_MAX_ACTIVE_CAMPAIGNS_PER_WORKSPACE=1
SCOUT_MAX_ACTIVE_SENDER_LANES=12
SCOUT_MAX_ACTIVE_SENDER_LANES_PER_WORKSPACE=2
SCOUT_SENDER_PAGE_SIZE=25
NEXT_PUBLIC_CENTRAL_WORKER_ENABLED=true

Keep:
GMAIL_SEND_ENABLED=true
GMAIL_REPLY_SYNC_ENABLED=false
GMAIL_NATIVE_SIGNATURE_SYNC_ENABLED=false
DELIVERABILITY_CENTER_ENABLED=true
SENDER_HEALTH_ENFORCEMENT_ENABLED=true
PLACEMENT_TESTS_ENABLED=true
TEAM_PAGINATION_ENABLED=true
ACCOUNT_DELETION_ENABLED=true

Keep all existing Supabase, Google OAuth, administrator, app-domain and support-email
variables.

STEP 4 — DEPLOY FROM WINDOWS GIT BASH
Put these in Downloads:
- scout-app-v10-35-1-scale-guard-r3.zip
- DEPLOY_V10_35_1_SCALE_GUARD_R3_GIT_BASH.sh

Run:
bash ~/Downloads/DEPLOY_V10_35_1_SCALE_GUARD_R3_GIT_BASH.sh

When prompted, type:
SCALE R3 SQL DONE

STEP 5 — WAIT FOR VERCEL READY
Confirm the deployment is Ready and login, Businesses, Templates, Messages, Settings,
Team, Upload and existing Replies history load.

STEP 6 — CREATE THE CENTRAL RENDER WORKER
Follow RENDER_SCALE_GUARD_WORKER_SETUP_V10_35_1.txt. Use the exact same
SCHEDULE_WORKER_SECRET in Vercel and Render.

STEP 7 — CONTROLLED TEST
1. Sign in as administrator.
2. Confirm connected Gmail accounts are paginated by 25.
3. Search for one active Gmail account.
4. Send one message to an inbox you control.
5. Confirm exactly one Scout signature.
6. Run six recipients with two Gmail accounts.
7. Close Scout and confirm Render continues the job.
8. Confirm Sent Today, rolling 24-hour usage and lifetime totals.
9. Confirm historical sent records remain visible.

ROLLBACK
The deployment script creates and pushes:
backup-before-v10-35-1-YYYYMMDD-HHMMSS

Restore that branch to main and redeploy Vercel if code rollback is required. The R3
SQL is additive and can remain; it does not activate work without the v10.35.1 code.
